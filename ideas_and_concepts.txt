инспирирован моим проектом vs_spice_solver - программа восстановления
электронной схемы по снятому ВАХ в точке схемы специальной 
измерительной установки. 

А) Основные проблемы vs_spice_solver -
    1. Медленная сходимость к решению(уточнение номиналов схемы ) при применении пакета
    оптимизации scipy.optimize и одномерной функции функции сравнения 
    кривых ВАХ.
    2. Трудности с определением первоначальной конфигурации электронной схемы.
    3. Представленные экспериментальные данные несоотвествовали электротехнике - 
    сильный дрейф и смещения нуля, фактически отрицательные сопротивления, гистерезис 
    в безгистерезисных схемах и проч.  
    4. Низкое качество кода - код писался в условии неясности концепции
    и большого дефицита времени. Фактически код одноразовых вычислений, приспособленный для повторного вызова.

Б) Чего мы хотим получить -
    1. Как минимум эквивалентую функциональность.
    2. Нормальное качество кода, доступное для экспериментов.
    3. ООП парадигма для схем. Базовая схема->Конкретная реализация схемы.
    Номиналы электронной схемы определяются по частным случаям конкретной реализации.
    4. Сохранение всех результатов моделирования(каждого вызова) ngspice,
    с целью сбора данных для обучения нейросети в дальнейшем.
    5. Нейросеть для предсказания принципиальной схемы.
    6. а) одномерная функция меры для сравнения ВАХ.
       б) многомерная функция и оптимизация номиналов на ее основе.
 
 В) Примерный план действий -
    1. Разработка на основе макетирования через TDD. Макет вычислений на основе мат вычислений scipy для 
    скорости.
    2. этапы - 
        2.1 одномерная оптимизация через перебор схем, набор статистики для ML.
        2.2 разработка нейросети для предсказания класса схемы.
        2.3 2.1+2.2 +> одномерная оптимизация с выбором схемы сразу нейросетью.
        2.4 2.3 +>выбор начальных параметров схемы для оптимизации нейросетью.
        2.5 оптимизация параметров на основе генеративно-состязательной нейросети.
    3. Замена макета вычислений на реальные вычисления в ngspice.
    4. все пункты 2.1-2.5 включительно.
